8. IAP/FREISCHALT-LOGIK: STRUKTURELLER MISMATCH
--------------------------------------------------------------------------------
Das ist der inhaltlich wichtigste Befund für die Businesslogik.

constants/products.ts definiert 4 separate kaufbare Module:
  { code: '101', title: 'Unit 1 Basics',  priceLabel: '5,99 €' },
  { code: '102', title: 'Unit 2 Urlaub',  priceLabel: '5,99 €' },
  { code: '103', title: 'Unit 3 Job',     priceLabel: '5,99 €' },
  { code: '104', title: 'Unit 4 Expat',   priceLabel: '5,99 €' },

constants/chapterUnlock.ts hat aber nur EINEN Purchase-Key:
  export const DISC101_FULL_KEY = 'purchase_disc101_full';

  export function isChapterLockedWithoutPurchase(chapterId: number): boolean {
    return chapterId > 1;   // gilt für ALLE 4 Units
  }

Das bedeutet: Ein einziger Kauf (disc101_full) schaltet Kapitel 2–7 in allen
4 Units frei. Es gibt keine separaten Kaufkeys für Units 102–104.

Mögliche Szenarien:
  A) ABSICHTLICH: Das Geschäftsmodell ist "eine Zahlung = alles", die 4
     Preisschilder sind nur Platzhalter. Dann sollte die UI klarstellen, dass
     ein Kauf alle Units freischaltet.

  B) UNFERTIG: Per-Unit-Käufe sind geplant, aber chapterUnlock.ts wurde noch
     nicht für Units 102–104 erweitert. Dann fehlen separate Keys und
     Prüflogik pro Modul.

Empfehlung: Diese Frage VOR dem App-Store-Launch mit dem Entwickler klären.
Das ist keine technische Fehlfunktion, aber eine offene Designentscheidung
mit direkter Auswirkung auf die Monetarisierung.


9. TEST-LOGIK NACH JEDEM KAPITEL — VOLLSTÄNDIGE ANALYSE
--------------------------------------------------------------------------------
Die Test-Logik ist VORHANDEN UND FUNKTIONAL.

Ablauf:
  1. Nutzer lernt alle Phrasen eines Kapitels durch Wischen
  2. Nach der letzten Phrase: Kapitel-Abschluss-Screen ("Kapitel abgeschlossen!")
  3. Zwei Buttons: "Tests starten" und "Nächstes Kapitel →"
  4. "Tests starten" öffnet Auswahl zwischen Test 1 und Test 2

Test 1 — Hören → Englisch erkennen
  Chris/Ann liest einen Satz vor. Nutzer wählt den gehörten Satz aus 3
  englischen Optionen.

Test 2 — Hören → Deutsch erkennen
  Chris/Ann liest einen Satz vor. Nutzer findet die deutsche Übersetzung
  aus 3 Optionen.

Feedback bei Abschluss:
  100% richtig     → Confetti + Audio "chris-das-war-spitze"
  ≥ 80% richtig    → Audio "ann-gut-gemacht" + rotierende Discokugel
  < 80%            → kein Sound, keine Animation
  Falsche Antworten → automatisch in den Repeat-Stapel gepinnt (sehr gut!)

--

BEFUND 9.1 — Test ist auf 8 Phrasen gedeckelt

  app/(tabs)/index.tsx:601-602
    const shuffled = shufflePhrases(chPhrases);
    const limited = shuffled.slice(0, 8);   // maximal 8 Phrasen

  Für kleine Kapitel wird alles abgefragt. Für große Kapitel ist die
  Abdeckung lückenhaft:

  -------   -------   -------   ---------
  Kap. 1    15        8         53%
  Kap. 2    12        8         67%
  Kap. 3    20        8         40%   ← niedrigste Abdeckung
  Kap. 4    18        8         44%
  Kap. 5    15        8         53%
  Kap. 6    8         8         100%
  Kap. 7    13        8         62%

  Kein Bug — kann eine bewusste UX-Entscheidung sein (kurze Tests = weniger
  Frustration). Pädagogisch werden in Kap. 3 und 4 aber mehr als die Hälfte
  der Phrasen nie getestet.

  Empfehlung zur Diskussion: Cap auf 10-12 erhöhen, oder adaptiv gestalten
  (alle Phrasen aus kleinen Kapiteln, Sample aus großen).

10. IRREFÜHRENDES "PRELOADING" IN utils/audioPreloader.ts
--------------------------------------------------------------------------------

  utils/audioPreloader.ts:16-28
    export function preloadPhraseAudio(phraseIds: number[], moduleCode: ModuleCode): void {
      for (const phraseId of phraseIds) {
        const mSrc = audioAssets[mKey];
        if (mSrc != null) preloadedCache[mKey] = mSrc;   // kopiert nur eine Referenz
      }
    }

Das "Preloading" kopiert lediglich Referenzen von audioAssets (einem statischen
Objekt) in preloadedCache. Da Expo-Audio-Assets via require() im Bundle liegen,
passiert kein echtes Vorausladen von Audiodaten. Die Funktion ist funktional
harmlos (schadet nicht), aber der Name preloadPhraseAudio suggeriert eine
Optimierung die nicht stattfindet.

Empfehlung: Entweder echtes Preloading implementieren (AudioPlayer vorab
initialisieren) oder die Funktion umbenennen und den Kommentar korrigieren.
